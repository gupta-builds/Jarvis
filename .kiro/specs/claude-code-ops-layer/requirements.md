# Requirements Document

## Introduction

Jarvis has a well-built Claude Code infrastructure: 11 skills, 4 agents, MCP servers, dashboards, a Capability Engine, and operating contracts. What it lacks is a daily operations layer — the thing that ties skills into cadences, monitors vault health automatically, detects drift before it compounds, and makes Claude Code the proactive manager of the vault rather than a reactive tool.

This feature turns Claude Code into the daily operations manager for Jarvis. It works through three mechanisms: a central dispatcher skill that orchestrates existing skills into coherent workflows, a set of automated monitoring checks that surface problems without being asked, and a daily cadence that connects morning planning to evening review through the Capability Engine.

The user runs Claude Code through OpenRouter (free tier), not direct Anthropic API. This means the operations layer must work within Claude Code's real extensibility surface — skills, agents, MCP servers, and CLAUDE.md — without assuming features that don't exist (no plugins marketplace, no linter hooks, no smart-connections).

## Glossary

- **Ops_Dispatcher**: A Claude Code skill (`.claude/skills/ops.md`) that acts as the central entry point for vault operations, routing to existing skills and running composite workflows.
- **Health_Check**: A structured scan of vault state that produces a machine-readable health report covering frontmatter drift, orphan notes, stale projects, inbox backlog, and Capability Engine gaps.
- **Drift_Detector**: The component of a Health_Check that compares current vault state against the canonical schema defined in `Vault Operating System.md` and `.obsidian/types.json`.
- **Daily_Cadence**: The morning-to-evening operational workflow that connects `/today` planning, mid-day triage, and `/closeday` review into a single coherent cycle.
- **Vault_Monitor**: The aggregate monitoring system composed of Health_Checks, Drift_Detection, and Capability Engine audits.
- **Capability_Audit**: A scan of the Capability Engine's state — overdue drills, notes missing evidence, stalled outputs, empty question banks — that feeds into the daily and weekly review cycles.
- **Triage_Queue**: A prioritized list of vault maintenance actions (fix broken links, process inbox items, update stale frontmatter) generated by the Vault_Monitor and presented during daily planning.
- **Session_Log**: The continuity record at `60_Claude/10_Session_Logs/log.md` that all operations append to.
- **Ops_Report**: A structured markdown report generated by the Vault_Monitor, stored in `60_Claude/50_Reviews/`.
- **Existing_Skills**: The 11 skills already defined in `.claude/skills/` — closeday, today, weekly-review, context, connect-notes, distill-note, ingest-clipping, lint-claude-layer, organize-csci2033, remove-ai-slop, trace-topic.

## Requirements

### Requirement 1: Central Operations Dispatcher

**User Story:** As a vault operator, I want a single skill that orchestrates daily vault operations, so that I don't have to remember which skills to run in which order.

#### Acceptance Criteria

1. WHEN the user invokes `/ops`, THE Ops_Dispatcher SHALL present a menu of available operations: morning-start, health-check, triage, ingest-batch, capability-audit, evening-close, and full-cycle.
2. WHEN the user selects "morning-start", THE Ops_Dispatcher SHALL execute `/context` followed by `/today`, then run a Health_Check, and present the combined output as a single morning briefing.
3. WHEN the user selects "evening-close", THE Ops_Dispatcher SHALL execute `/closeday`, run a Capability_Audit, and append both results to the Session_Log.
4. WHEN the user selects "full-cycle", THE Ops_Dispatcher SHALL execute morning-start, wait for user work, then execute evening-close when re-invoked.
5. WHEN the user selects "ingest-batch", THE Ops_Dispatcher SHALL list all unprocessed files in `60_Claude/05_Clippings/`, let the user select which to process, and run `/ingest-clipping` for each selected file sequentially.
6. IF the Ops_Dispatcher encounters an error during any sub-skill execution, THEN THE Ops_Dispatcher SHALL log the error to the Session_Log, skip the failed step, and continue with remaining operations.
7. THE Ops_Dispatcher SHALL read `AI_CONTEXT.md` and `HUMAN_WRITING.md` before generating any prose output.

### Requirement 2: Vault Health Check System

**User Story:** As a vault operator, I want automated health checks that detect problems across the entire vault, so that drift and decay are caught before they compound.

#### Acceptance Criteria

1. WHEN the user invokes `/ops health-check` or the Health_Check runs as part of a composite workflow, THE Vault_Monitor SHALL scan the vault across six dimensions: frontmatter completeness, link integrity, inbox backlog, project staleness, Capability Engine gaps, and Claude layer health.
2. THE Drift_Detector SHALL compare every note's frontmatter against the canonical schema in `Vault Operating System.md` and report notes missing required fields (`type`, `status`, `created`, `tags`).
3. THE Vault_Monitor SHALL identify notes with broken wikilinks — links pointing to notes that do not exist in the vault.
4. THE Vault_Monitor SHALL count unprocessed items in `00_Inbox/` and `60_Claude/05_Clippings/` and flag when either exceeds 10 items.
5. THE Vault_Monitor SHALL identify projects in `20_Progress/` that have no `next:` action or have not been modified in 30 or more days.
6. THE Vault_Monitor SHALL produce an Ops_Report in `60_Claude/50_Reviews/Ops Health - YYYY-MM-DD.md` with a summary table, per-dimension findings, and a prioritized Triage_Queue.
7. IF the Health_Check finds zero issues across all dimensions, THEN THE Vault_Monitor SHALL report a clean bill of health with the scan timestamp.

### Requirement 3: Capability Engine Audit

**User Story:** As a vault operator, I want the operations layer to monitor the Capability Engine's health, so that overdue drills, missing evidence, and stalled outputs are surfaced during daily operations.

#### Acceptance Criteria

1. WHEN the Capability_Audit runs, THE Vault_Monitor SHALL query all notes with a `track` field and report: total tracked concepts, count by mastery_level, count of notes with overdue `next_drill` dates, and count of notes with empty `evidence` fields.
2. WHEN the Capability_Audit runs, THE Vault_Monitor SHALL check each track's Question Bank for tracks with fewer than 3 open questions and flag them as underseeded.
3. WHEN the Capability_Audit runs, THE Vault_Monitor SHALL check `60_Claude/45_Outputs/` for output notes with `status: seed` older than 14 days and flag them as stalled.
4. THE Capability_Audit SHALL produce a "Capability Health" section within the Ops_Report that includes: drill queue (top 5 overdue), evidence gaps (top 5 concepts missing evidence), stalled outputs, and underseeded question banks.
5. WHEN the Capability_Audit identifies overdue drills, THE Vault_Monitor SHALL sort them by days overdue descending and include the note title, track, and days overdue in the report.

### Requirement 4: Daily Operations Cadence

**User Story:** As a vault operator, I want a structured daily cadence that connects morning planning to evening review through the Capability Engine, so that each day has a coherent operational rhythm.

#### Acceptance Criteria

1. WHEN the morning-start operation runs, THE Ops_Dispatcher SHALL produce a "Morning Briefing" note in `60_Claude/50_Reviews/` that combines: today's plan (from `/today`), vault health summary (from Health_Check), top 3 triage items, and top 3 overdue drills from the Capability_Audit.
2. WHEN the evening-close operation runs, THE Ops_Dispatcher SHALL produce an "Evening Close" section appended to the day's Closeday note that includes: Capability_Audit summary, triage items completed vs remaining, and a one-line vault health delta compared to the morning check.
3. THE Daily_Cadence SHALL ensure that every morning briefing links to the previous day's closeday note using wikilinks, creating a continuous chain.
4. WHEN the user runs `/ops` without specifying an operation, THE Ops_Dispatcher SHALL detect the time of day and suggest "morning-start" before noon and "evening-close" after noon.
5. THE Daily_Cadence SHALL append a structured entry to the Session_Log after both morning-start and evening-close operations complete.

### Requirement 5: Triage Queue Generation

**User Story:** As a vault operator, I want the operations layer to generate a prioritized list of maintenance actions, so that I can work through vault upkeep systematically instead of ad hoc.

#### Acceptance Criteria

1. WHEN the Vault_Monitor generates a Triage_Queue, THE Vault_Monitor SHALL assign each item a priority: critical (broken links, missing required frontmatter), high (stale projects, overdue drills), medium (inbox backlog, unprocessed clippings), or low (weak links, orphan notes).
2. THE Triage_Queue SHALL contain no more than 15 items, selecting the highest-priority items when more exist.
3. WHEN the user selects "triage" from the Ops_Dispatcher menu, THE Ops_Dispatcher SHALL present the current Triage_Queue and offer to execute fixes for each item, routing to the appropriate Existing_Skill where one exists.
4. WHEN a triage item maps to an existing skill (broken links to `/connect-notes`, clippings to `/ingest-clipping`, Claude layer issues to `/lint-claude-layer`), THE Ops_Dispatcher SHALL invoke that skill with the relevant parameters.
5. THE Triage_Queue SHALL persist in the Ops_Report so that items not addressed in one session carry forward to the next Health_Check.

### Requirement 6: Ops Dispatcher Skill File

**User Story:** As a vault operator, I want the operations dispatcher implemented as a Claude Code skill file, so that it integrates with the existing skill invocation pattern.

#### Acceptance Criteria

1. THE Ops_Dispatcher SHALL be implemented as `.claude/skills/ops.md` following the same structure as Existing_Skills (title, description, usage, instructions, constraints).
2. THE Ops_Dispatcher SHALL reference Existing_Skills by their slash-command names rather than duplicating their logic.
3. THE Ops_Dispatcher SHALL include a "Safety Constraints" section that enforces: read `AI_CONTEXT.md` before operations, never modify `60_Claude/05_Clippings/`, preserve frontmatter on all edits, and search before creating notes.
4. WHEN the Ops_Dispatcher creates or modifies vault notes, THE Ops_Dispatcher SHALL follow the frontmatter conventions defined in `CLAUDE.md` and `Vault Operating System.md`.
5. THE Ops_Dispatcher SHALL include usage examples for each operation mode in its skill file documentation.

### Requirement 7: Vault Curator Agent Enhancement

**User Story:** As a vault operator, I want the existing vault-curator agent enhanced to work as the deep-maintenance counterpart to daily ops, so that weekly and monthly maintenance is covered.

#### Acceptance Criteria

1. WHEN the user invokes the vault-curator agent, THE vault-curator SHALL read the most recent Ops_Report to avoid re-scanning issues already identified by the daily Health_Check.
2. THE vault-curator SHALL add a "Capability Engine Maintenance" section to its Curator Report that covers: notes with conflicting property types, tracks with no enriched notes in 30 days, and synthesis notes with broken concept links.
3. WHEN the vault-curator identifies duplicate notes, THE vault-curator SHALL check whether either note has Capability Engine fields (`track`, `mastery_level`, `evidence`) and prefer preserving the enriched version.
4. THE vault-curator agent file SHALL be updated at `.claude/agents/vault-curator.md` to include these enhanced behaviors while preserving all existing functionality.

### Requirement 8: Operations Report Format

**User Story:** As a vault operator, I want operations reports in a consistent, scannable format, so that I can quickly assess vault health and decide what to act on.

#### Acceptance Criteria

1. THE Ops_Report SHALL use this structure: Summary Table (dimension, status emoji, count), Frontmatter Drift section, Link Health section, Inbox/Clippings Backlog section, Project Health section, Capability Engine Health section, and Triage Queue section.
2. THE Ops_Report SHALL use frontmatter with `type: review`, `status: complete`, `created: YYYY-MM-DD`, and tags including `ops-health` and `review`.
3. THE Ops_Report SHALL use status emojis: ✅ for healthy (zero issues), ⚠️ for warning (1-5 issues), ❌ for critical (6+ issues or any broken required fields).
4. WHEN the Ops_Report includes a Triage_Queue, THE Ops_Report SHALL format each item as a task checkbox with priority tag, description, and suggested skill command.
5. THE Ops_Report SHALL include a "Comparison" section when a previous Ops_Report exists, showing which dimensions improved, worsened, or stayed the same.

### Requirement 9: Session Log Integration

**User Story:** As a vault operator, I want all operations layer activity logged to the session log, so that continuity is maintained across sessions and tools.

#### Acceptance Criteria

1. WHEN any Ops_Dispatcher operation completes, THE Ops_Dispatcher SHALL append a structured entry to `60_Claude/10_Session_Logs/log.md` with: date, operation name, files created or modified, and a one-line summary.
2. THE Session_Log entry format SHALL follow the existing convention: `## [YYYY-MM-DD] ops | [operation-name]` followed by bullet points for created files, modified files, and key findings.
3. WHEN the morning-start operation reads the Session_Log for context, THE Ops_Dispatcher SHALL extract the most recent 5 entries and include relevant carryover items in the morning briefing.
4. IF the Session_Log file does not exist, THEN THE Ops_Dispatcher SHALL create it with appropriate frontmatter before appending.

### Requirement 10: CLAUDE.md Contract Update

**User Story:** As a vault operator, I want CLAUDE.md updated to document the operations layer, so that the operating contract reflects the new capabilities.

#### Acceptance Criteria

1. THE CLAUDE.md update SHALL add `/ops` to the Available Skills table with a description of "Daily vault operations — health checks, triage, cadence management".
2. THE CLAUDE.md update SHALL add a "Daily Operations Cadence" section under Core Rules that describes the morning-start → work → evening-close rhythm.
3. THE CLAUDE.md update SHALL preserve all existing content, frontmatter, and formatting in CLAUDE.md.
4. WHEN the CLAUDE.md update adds new content, THE update SHALL keep the total file under 200 lines by linking to the ops skill file for detailed documentation rather than inlining it.

### Requirement 11: OpenRouter Compatibility

**User Story:** As a vault operator using OpenRouter's free tier, I want the operations layer designed for efficiency, so that it works within the constraints of routed model access.

#### Acceptance Criteria

1. THE Ops_Dispatcher SHALL minimize MCP tool calls by batching vault reads where possible — reading multiple files in a single context-gathering pass rather than one file per operation.
2. THE Ops_Dispatcher SHALL structure Health_Check scans to use Obsidian MCP search operations rather than reading every file individually.
3. WHEN the Ops_Dispatcher generates reports, THE Ops_Dispatcher SHALL prefer concise, structured output (tables, bullet lists) over verbose prose to reduce token usage.
4. THE Ops_Dispatcher skill file SHALL document which operations are lightweight (health-check, triage) and which are heavyweight (full-cycle, ingest-batch) so the user can choose based on available context budget.
5. IF an MCP tool call fails due to a connection or timeout error, THEN THE Ops_Dispatcher SHALL retry once, then log the failure and continue with remaining operations.

### Requirement 12: Jarvis Ops CLI Integration

**User Story:** As a vault operator, I want the Claude Code operations layer to reuse the local Jarvis Ops CLI, so that expensive model-driven scans do not repeat work that deterministic tooling can do cheaply.

#### Acceptance Criteria

1. WHEN `/ops health-check` runs in a local shell-capable environment, THE Ops_Dispatcher SHALL prefer the deterministic CLI at `30_Order/System/jarvis-cli/jarvis.ps1 health` for baseline counts before doing model interpretation.
2. WHEN `/ops context` or morning-start runs, THE Ops_Dispatcher SHALL use `30_Order/System/jarvis-cli/jarvis.ps1 context` as the first-pass context pack, then read only the additional notes needed for the selected operation.
3. WHEN `/ops report` or health-check creates a report, THE Ops_Dispatcher SHALL align the report with the CLI's current scan dimensions: metadata gaps, future dates, project next actions, duplicate filenames, link health, and likely encoding damage.
4. WHEN the CLI report path differs from the Ops_Report path required by this spec, THE Ops_Dispatcher SHALL link both reports together rather than duplicating full findings.
5. IF the CLI is unavailable or fails, THEN THE Ops_Dispatcher SHALL fall back to MCP/search-based checks, log the fallback reason, and continue.
6. THE Ops_Dispatcher SHALL treat CLI output as diagnostic input, not as permission to automatically rewrite notes.

### Requirement 13: Time and Date Consistency

**User Story:** As a vault operator, I want the operations layer to detect impossible or future-dated metadata, so that dashboards do not treat planned or mistaken dates as real freshness.

#### Acceptance Criteria

1. WHEN any Health_Check runs, THE Vault_Monitor SHALL compare `created`, `updated`, `reviewed`, `last_drilled`, and `next_drill` fields against the actual current date.
2. WHEN a note has an `updated` date in the future, THE Vault_Monitor SHALL flag it as metadata drift because it can make stale notes look fresh.
3. WHEN a `next_drill` date is in the future, THE Vault_Monitor SHALL not treat it as an error by itself; it SHALL only flag it if the paired `last_drilled` date is also in the future or inconsistent.
4. WHEN the Session_Log contains entries dated after the actual current date, THE Vault_Monitor SHALL surface a "timeline inconsistency" finding in the Ops_Report.
5. THE Ops_Report SHALL include an explicit "Date Consistency" subsection listing the top future-dated fields and distinguishing scheduled future work from impossible history.

### Requirement 14: Encoding and Text Integrity Audit

**User Story:** As a vault operator, I want the operations layer to detect mojibake and text corruption, so that durable notes do not slowly fill with unreadable characters.

#### Acceptance Criteria

1. WHEN Health_Check runs, THE Vault_Monitor SHALL scan for likely mojibake sequences such as broken smart punctuation, broken arrows, and UTF-8 marker artifacts.
2. THE Vault_Monitor SHALL rank encoding findings by hit count and folder sensitivity, prioritizing active notes before archived notes.
3. THE Vault_Monitor SHALL not auto-repair encoding damage unless the user explicitly requests a targeted cleanup batch.
4. WHEN encoding damage appears in `50_Archive/` or `60_Claude/05_Clippings/`, THE Vault_Monitor SHALL report it as low priority unless the damaged file is actively linked from a current dashboard or project.
5. WHEN encoding damage appears in operating contracts, dashboards, templates, or active project notes, THE Vault_Monitor SHALL mark it as high priority because it affects daily retrieval and AI behavior.

### Requirement 15: Scope Boundaries and Safe Mutation Policy

**User Story:** As a vault operator, I want daily ops to be safe by default, so that automation can inspect the vault without unexpectedly changing source material.

#### Acceptance Criteria

1. THE Ops_Dispatcher SHALL separate "scan", "suggest", and "fix" modes; scan and suggest modes SHALL be read-only.
2. THE Ops_Dispatcher SHALL never modify `60_Claude/05_Clippings/` unless the user explicitly asks for raw clipping cleanup.
3. THE Ops_Dispatcher SHALL treat `50_Archive/` as read-only historical reference unless the selected operation is archival cleanup.
4. THE Ops_Dispatcher SHALL not edit `.obsidian` plugin data, MCP credential files, or local settings files during normal health checks.
5. WHEN a fix touches more than 5 notes, THE Ops_Dispatcher SHALL create or present a batch plan before editing.
6. WHEN a fix would affect a note with `status: tree`, THE Ops_Dispatcher SHALL present the proposed change before applying it.
7. THE Ops_Dispatcher SHALL prefer appending reports and patching narrow sections over rewriting whole notes.

### Requirement 16: Kiro Spec and Implementation Handoff

**User Story:** As a vault operator using Kiro for planning, I want the operations layer to keep Kiro specs, requirements, and implementation tasks connected, so that planning work becomes executable without losing context.

#### Acceptance Criteria

1. WHEN a Kiro spec exists under `.kiro/specs/`, THE Ops_Dispatcher SHALL treat its `requirements.md`, `design.md`, and `tasks.md` files as planning artifacts, not ordinary vault notes.
2. WHEN `/ops health-check --include-tools` or the equivalent operation runs, THE Vault_Monitor SHALL include Kiro specs in the tool-layer audit and report stale or incomplete specs.
3. WHEN an Ops_Report references a Kiro spec, THE report SHALL link to the specific spec folder and summarize whether it has requirements, design, and tasks files.
4. WHEN Kiro-generated plans mention implementation work already covered by local tooling, THE Ops_Dispatcher SHALL point to the local tool instead of duplicating the implementation.
5. THE Ops_Dispatcher SHALL not mutate `.kiro/specs/` during normal daily ops unless the user explicitly asks to update a Kiro plan.
6. WHEN the user asks to improve a Kiro plan, THE agent SHALL add content to the relevant spec and preserve existing text unless explicitly told to refactor.

### Requirement 17: Report Registry and Carry-Forward State

**User Story:** As a vault operator, I want ops reports to carry state forward across days, so that recurring issues do not disappear just because yesterday's report scrolled away.

#### Acceptance Criteria

1. THE Vault_Monitor SHALL maintain a lightweight report index or use an existing dashboard to surface the most recent Ops_Report, Morning Briefing, Evening Close, and CLI-generated ops report.
2. WHEN generating a new Ops_Report, THE Vault_Monitor SHALL compare against the most recent prior Ops_Report and mark recurring items as "carried forward".
3. WHEN a triage item appears in three consecutive Ops_Reports, THE Vault_Monitor SHALL raise its priority by one level unless it is explicitly marked deferred.
4. WHEN a triage item is resolved, THE next Ops_Report SHALL include it in a short "resolved since last report" list.
5. THE Triage_Queue SHALL include stable identifiers derived from category, path, and issue type so repeated issues can be matched across reports.

### Requirement 18: Dashboard and Base Integration

**User Story:** As a vault operator, I want ops findings to show up in Obsidian dashboards and Bases, so that the daily operations layer is visible outside Claude Code.

#### Acceptance Criteria

1. WHEN the operations layer creates recurring reports, THE implementation SHALL add or update a dashboard entry that links to the latest Ops_Report without manually duplicating all findings.
2. THE operations layer SHALL prefer Dataview queries for dynamic dashboards and Bases for editable registries, matching the existing Capability Engine pattern.
3. IF a new Base file is added for operations, THEN it SHALL live under `60_Claude/60_Indexes/Bases/` and track report status, created date, top issue count, and carry-forward count.
4. THE operations layer SHALL not require unsupported plugins beyond the vault's current enabled set: Dataview, Tasks, Templater, Local REST API, Bases, and existing navigation/style helpers.
5. WHEN a dashboard query depends on a new metadata field, THE implementation SHALL document that field in `Vault Operating System.md` or an ops-specific system note.

### Requirement 19: Raw Capture to Distillation Throughput

**User Story:** As a vault operator, I want daily ops to manage the capture backlog without turning every clipping into durable knowledge, so that ingestion stays selective and useful.

#### Acceptance Criteria

1. WHEN the clippings backlog exceeds the configured threshold, THE Triage_Queue SHALL group clippings by likely topic or folder rather than listing every file separately.
2. WHEN ingest-batch runs, THE Ops_Dispatcher SHALL distinguish quick summaries, standard distillations, and deep analyses before starting the batch.
3. THE Ops_Dispatcher SHALL process clippings sequentially and stop after a user-configurable batch limit to protect context budget.
4. WHEN a clipping has already produced a source summary, THE Ops_Dispatcher SHALL mark it as already processed and avoid duplicate summaries.
5. WHEN a source summary produces durable knowledge, THE Ops_Dispatcher SHALL route it through the existing promotion logic: source summary first, then concept/project/output only when the idea is reusable.

### Requirement 20: Knowledge Graph Quality and Duplicate Policy

**User Story:** As a vault operator, I want link and duplicate checks to respect the vault's structure, so that course notes, templates, and durable notes are not judged by the same rules.

#### Acceptance Criteria

1. THE Vault_Monitor SHALL classify duplicate filenames by risk: high for durable notes, medium for active project notes, low for course-week or archive patterns where path context is expected.
2. THE Vault_Monitor SHALL treat template placeholder wikilinks differently from real broken links.
3. WHEN a duplicate pair includes one enriched Capability Engine note and one feeder-layer course note, THE triage recommendation SHALL prefer linking the feeder note to the enriched note rather than merging them.
4. WHEN broken links point to likely missing concepts, THE Triage_Queue SHALL suggest `/connect-notes` or `/distill-note` rather than creating empty stubs automatically.
5. WHEN broken links point to attachments, THE Vault_Monitor SHALL check common attachment locations before labeling them as missing notes.

### Requirement 21: Cost and Context Budget Modes

**User Story:** As a vault operator using routed model access, I want each ops workflow to declare its cost profile, so that I can choose a lightweight or heavyweight pass intentionally.

#### Acceptance Criteria

1. THE Ops_Dispatcher SHALL label operations as lightweight, standard, or heavyweight in `.claude/skills/ops.md`.
2. Lightweight operations SHALL rely on CLI summaries, Dataview-style counts, and short report sections.
3. Standard operations SHALL read the live context files plus only the notes referenced by the current triage queue.
4. Heavyweight operations SHALL be reserved for ingest-batch, full-cycle, curator review, and deep duplicate analysis.
5. WHEN context budget is constrained, THE Ops_Dispatcher SHALL prefer producing a report with paths and counts over reading full note bodies.
6. THE Ops_Dispatcher SHALL include a "next best lightweight command" in failure messages when a heavyweight operation cannot complete.

### Requirement 22: Verification and Acceptance Harness

**User Story:** As a vault operator, I want a concrete verification checklist for the operations layer, so that I can tell whether the system works beyond just creating files.

#### Acceptance Criteria

1. THE implementation SHALL verify that `/ops health-check` or its local equivalent produces an Ops_Report with the required sections.
2. THE implementation SHALL verify that `/ops morning-start` creates a Morning Briefing and appends a Session_Log entry.
3. THE implementation SHALL verify that `/ops evening-close` updates or links to the day's closeday note and appends a Session_Log entry.
4. THE implementation SHALL verify that report generation does not modify `60_Claude/05_Clippings/`, `50_Archive/`, `.obsidian`, or credential-bearing MCP config files.
5. THE implementation SHALL verify that the ops layer still works when Obsidian MCP is unavailable by falling back to local file scanning where possible.
6. THE implementation SHALL document the final command sequence used for verification in the Session_Log.

## Additive Implementation Notes

These notes do not replace the requirements above. They clarify how the current Jarvis vault should influence implementation.

- The first deterministic baseline already exists at `30_Order/System/jarvis-cli/`. The Claude skill should call or mirror it rather than inventing a second scanner.
- Current default CLI baseline after the first report: 628 Markdown files scanned, 147 future-dated metadata fields, 10 project notes, 7 active projects missing `next`, 807 broken wikilinks, and 99 ambiguous wikilinks.
- The existing report location is `60_Claude/50_Reviews/Ops Reports/`; the Kiro requirements can still create `60_Claude/50_Reviews/Ops Health - YYYY-MM-DD.md`, but the two report families should cross-link.
- The current vault has a real time-consistency issue: several notes and session-log entries are dated after the actual date of 2026-04-24. Ops should surface this instead of treating those entries as reliable current state.
- The enabled plugin reality matters: Dataview, Tasks, Templater, Local REST API, and Bases are available; Smart Connections and hypothetical hooks should not be assumed.
- The operations layer should keep `AI_CONTEXT.md`, `00_Dashboard.md`, and `60_Claude/10_Session_Logs/log.md` as the live context spine. Tool-specific files should point back to that spine instead of becoming parallel memories.
- Reports should be short enough to scan. The point is to create decisions and next actions, not a second vault made of audit prose.
