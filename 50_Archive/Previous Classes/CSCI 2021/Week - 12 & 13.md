### 6.1 Storage Technologies
This section outlines the physical characteristics and performance trade-offs of the hardware devices that store data.

6.1.1 Random Access Memory (RAM)

RAM is divided into two types:

1. **Static RAM (SRAM):**

    ◦ **Used for:** Cache memories (L1, L2, L3).

    ◦ **Characteristics:** **Faster** than DRAM, significantly **more expensive**, and uses more transistors per bit (lower density). It is persistent as long as power is applied and **does not need to be refreshed**.

2. **Dynamic RAM (DRAM):**

    ◦ **Used for:** Main memory and graphics frame buffers.

    ◦ **Characteristics:** **Slower** than SRAM, less expensive, and has higher density. It stores each bit as a charge on a capacitor and must be **periodically refreshed**.

    ◦ **Organization:** DRAM chips are organized internally as rectangular arrays of supercells (bits). To read data, the memory controller sends a **RAS (Row Access Strobe)** request followed by a **CAS (Column Access Strobe)** request. DRAM chips are packaged into **memory modules** (like DIMMs) to connect to the memory controller and move data in chunks (e.g., 64-bit words).

6.1.2 Disk Storage

Disks (rotating disks) provide huge storage capacity but are extremely slow compared to semiconductor memory.

• **Geometry:** Disks use spinning **platters** covered in magnetic material, divided into **tracks** (concentric rings) which are further segmented into **sectors** (fixed-size blocks, typically 512 bytes). **Cylinders** are the collection of tracks equidistant from the spindle across all surfaces.

• **Disk Access Time (**Taccess​**):** Accessing a sector is dominated by mechanical delay: Taccess​=Tavg seek​+Tavg rotation​+Tavg transfer​

    ◦ **Seek Time (**Tavg seek​**):** Time to move the read/write arm over the correct track (milliseconds).

    ◦ **Rotational Latency (**Tavg rotation​**):** Time to wait for the first bit of the sector to rotate under the head. Tavg rotation​≈1/2×Tmax rotation​.

    ◦ **Transfer Time (**Tavg transfer​**):** Time to read the bits in the sector; this component is very small/negligible.

    ◦ **Problem-Solving Focus:** Textbook questions often require calculating Taccess​ given RPM, sectors/track, and seek time.

• **Logical Blocks:** To hide the complex physical geometry, disks present a simpler view as a sequence of **logical blocks**, with a disk controller handling the translation to (surface, track, sector) triples.

• **I/O and DMA:** The CPU communicates with devices using **memory-mapped I/O**. Data transfer between the disk and main memory typically happens via **DMA (Direct Memory Access)**, which means the device performs the transfer directly without continuous CPU involvement.

6.1.3 Solid State Disks (SSDs)

SSDs use **flash memory** (a type of nonvolatile memory) instead of spinning platters. They use a **flash translation layer** to manage logical blocks.

• **Performance:** SSDs are significantly faster and more durable than rotating disks. **Reading is faster than writing** because writing requires erasing entire blocks before pages can be rewritten.

6.1.4 Storage Technology Trends

Since 1985, the cost per byte for DRAM and disks has decreased drastically, but **access times have improved much more slowly**. This increasing difference between CPU speed and memory/disk speed is the **processor-memory gap**. Modern systems rely on caches to bridge this gap.
## Chapter 5: Optimizing Program Performance (Sections 5.7 – 5.14)

These sections focus on **machine-dependent optimization**, emphasizing how to write code that maximizes efficiency by exploiting the capabilities of modern CPUs, especially their ability to execute operations in parallel.

5.7 Understanding Modern Processors

To push past the limits set by sequential programming, optimization must target the processor's architecture (microarchitecture).

1. **Superscalar and Out-of-Order Execution:** Modern CPUs are **superscalar**, meaning they can perform multiple operations every clock cycle, and **out-of-order**, meaning they execute instructions not necessarily in the program sequence, as long as dependencies allow. This capability is called **instruction-level parallelism (ILP)**.

2. **ICU and EU:** The CPU consists mainly of the **Instruction Control Unit (ICU)**, which fetches and decodes instructions into simpler operations, and the **Execution Unit (EU)**, which executes these operations.

3. **Speculative Execution:** The CPU uses **branch prediction** to guess the outcome of conditional jumps and starts executing instructions along the predicted path. If the guess is wrong, the speculative work is discarded, incurring a significant **misprediction penalty**.

4. **Functional Units:** The EU contains multiple **functional units** (e.g., ALU for arithmetic, load units, store units) that are specialized to handle different types of operations simultaneously.

5. **Performance Bounds:** Program performance is limited by two primary bounds:

    ◦ **Latency Bound:** The minimum time required due to a chain of sequential operations, where the result of one operation is immediately needed for the next (the length of the **critical path** in the data flow).

    ◦ **Throughput Bound:** The maximum rate at which the processor's functional units can sustain production, determined by their **capacity** (number of units) and **issue time** (minimum time between starting identical operations).

5.8 Loop Unrolling

**Loop Unrolling** is a transformation where the number of operations inside the loop body is increased, typically reducing the total number of iterations.

• **Benefits:** It primarily reduces the non-productive overhead associated with loop maintenance (indexing and branching).

• **Limitation (k x 1 Unrolling):** Simple unrolling (k x 1) does _not_ increase the utilization of parallel hardware because the bottleneck remains the sequential dependency of the single accumulator variable, meaning performance remains limited by the **latency bound**.

5.9 Enhancing Parallelism

These techniques explicitly restructure the code to break the critical dependency chain on a single accumulator, thus exploiting ILP.

1. **Multiple Accumulators (k x k Unrolling):** For associative and commutative operations (like addition or multiplication), the computation is split across multiple independent accumulator variables (e.g., `acc0`, `acc1`).

    ◦ This breaks the sequential dependency, effectively shortening the critical path by the unrolling factor k. This allows the program to approach the **throughput bound**.

2. **Reassociation Transformation (k x 1a Unrolling):** This involves merely moving parentheses to change the order of evaluation (e.g., `acc = acc OP (data[i] OP data[i+1])` vs. `acc = (acc OP data[i]) OP data[i+1]`).

    ◦ The internal operation (`data[i] OP data[i+1]`) can execute in parallel with the main dependency chain, effectively increasing parallelism and approaching the throughput bound, similar to using multiple accumulators.

    ◦ **Crucial Note:** Compilers are cautious and generally **do not** perform these parallelizing transformations (multiple accumulators or reassociation) on floating-point operations because floating-point arithmetic is **not strictly associative** (due to rounding). Programmers must perform these optimizations manually.

5.11 Some Limiting Factors

1. **Register Spilling:** If the program's parallelism requires holding more intermediate results than the number of available CPU registers, the compiler must resort to **spilling** these temporary values onto the stack (memory). Accessing memory is much slower than accessing registers, and this dramatically degrades performance.

2. **Branch Prediction and Misprediction Penalties:** Unpredictable conditional branches incur the heavy **misprediction penalty** (discarded speculative work). To avoid this, programmers should write code that encourages the compiler to use **conditional move** instructions (conditional data transfers) rather than conditional control transfers. This is often achieved by writing code in a **functional style** (using ternary operators).

5.12 Understanding Memory Performance

This focuses on memory performance when data is resident in the cache hierarchy.

1. **Load Performance:** Load operations (reading from memory to register) can be slowed by **load latency** when the address of one load operation depends on the result of a previous load operation (e.g., fetching nodes in a linked list).

2. **Store Performance (Write/Read Dependency):** Store operations (writing from register to memory) are complicated when followed closely by a load operation that reads the same memory location (**memory aliasing**). This **write/read dependency** forces the load to check the **store buffer** (a queue of pending writes) maintained by the store unit. If the addresses match, the load must wait for the store data, significantly slowing execution.

5.13 Life in the Real World: Performance Improvement Techniques

Optimization strategies involve multiple levels of abstraction:

1. **High-Level Design:** Selecting algorithms and data structures that provide low asymptotic complexity (e.g., using quicksort instead of insertion sort).

2. **Basic Coding Principles:** Avoiding optimization blockers (aliasing, procedure calls, loop invariants that change slowly).

3. **Low-Level Optimizations:** Restructuring loops using techniques like unrolling and multiple accumulators to expose ILP.

4. **Checking Code:** Using testing routines after every optimization step to ensure functional correctness.

5.14 Identifying and Eliminating Performance Bottlenecks

1. **Code Profilers (gprof):** Tools used to measure how much time a program spends executing different functions, helping programmers focus optimization efforts on the actual performance bottlenecks.

2. **Amdahl’s Law:** Reinforces that the overall speedup achieved by optimizing a system component is limited by the fraction of the total time that component was used.

Chapter 9: Virtual Memory (Sections 9.1 – 9.5)

These sections introduce the foundational concepts of Virtual Memory (VM), explaining the address translation mechanism and the three core capabilities VM provides.

9.1 Physical and Virtual Addressing

1. **Physical Addressing (PA):** The CPU generates a physical address that is sent directly to the main memory (DRAM) over the memory bus. This is the simplest model, used by older or embedded systems.

2. **Virtual Addressing (VA):** Modern CPUs use virtual addressing, generating a **Virtual Address (VA)**.

3. **Address Translation:** The VA is converted into a **Physical Address (PA)** by specialized hardware on the CPU chip called the **Memory Management Unit (MMU)**. This process requires cooperation between the hardware and the operating system (OS).

9.2 Address Spaces

1. **Address Space:** An ordered set of contiguous, nonnegative integer addresses.

2. **Virtual Address Space (VAS):** The set of N=2n virtual addresses generated by the CPU.

3. **Physical Address Space (PAS):** Corresponds to the M bytes of physical memory (DRAM).

4. **VM Function:** VM assigns each byte of main memory a VA (from VAS) and a PA (from PAS).

9.3 VM as a Tool for Caching

VM treats main memory as a large cache for the contents of the virtual address space, which is conceptually stored on disk.

1. **Virtual Pages (VPs) and Physical Pages (PPs):** Both virtual memory and physical memory are partitioned into fixed-size blocks called **Virtual Pages (VPs)** and **Physical Pages (PPs)** (or page frames), respectively. Pages are typically large (4 KB or more).

2. **VP States:** A virtual page can be **Unallocated** (not yet created), **Cached** (allocated and currently in physical memory), or **Uncached** (allocated but only resident on disk).

3. **DRAM Cache Organization:** The cache that stores VPs in main memory (DRAM cache) is typically **fully associative** (any VP can be stored in any PP) and uses a **write-back** policy due to the high cost of disk I/O.

4. **Page Tables:** The OS maintains a data structure called a **page table** in physical memory to manage the mapping between VPs and PPs.

5. **Page Table Entries (PTEs):** Each page in the VAS has a corresponding PTE. A PTE contains a **valid bit** (indicating if the VP is cached) and an address field (either the Physical Page Number (PPN) or the disk address).

6. **Page Hits:** If the valid bit is set (**page hit**), the MMU uses the PPN from the PTE to form the PA and retrieve the data.

7. **Page Faults:** If the valid bit is zero (**page miss** or **page fault**), the MMU triggers an exception. The kernel's **page fault handler** selects a **victim page**, pages the required VP into memory from disk (**demand paging**), updates the PTE, and restarts the instruction.

8. **Locality:** VM works well because programs usually exhibit good temporal locality, limiting their memory access to a small **working set** of pages.

9.4 VM as a Tool for Memory Management

VM simplifies memory management through the use of separate page tables for each process.

1. **Separate Address Spaces:** Each process is given its own independent virtual address space, enabled by the kernel maintaining a distinct page table for each process.

2. **Simplification:** This approach simplifies **linking** (all processes use the same memory format), **loading** (programs are loaded lazily via **memory mapping** only when pages are first referenced), and memory **sharing** (multiple VPs in different processes can point to the same PP).

9.5 VM as a Tool for Memory Protection

The address translation mechanism also enforces access control and protection.

1. **Isolation:** Separate address spaces naturally isolate the private memory of one process from others.

2. **Permission Bits:** **Permission bits** are added to the PTE (e.g., SUP for kernel/user mode, READ/WRITE permissions) to control access to specific pages.

3. **Protection Faults:** If an instruction violates these permissions (e.g., a user-mode process tries to write to a read-only page), the CPU triggers a **general protection fault** (segmentation fault).
9.6 Address Translation

Address translation is the hardware-supported process that maps a Virtual Address (VA) generated by the CPU to a Physical Address (PA) in DRAM.

Address Components and the MMU

The Memory Management Unit (MMU) is dedicated hardware on the CPU chip responsible for translating VAs on the fly.

1. **Virtual Address (VA) Structure:** The n-bit VA is split into two parts: the Virtual Page Number (VPN) and the Virtual Page Offset (VPO).

2. **Physical Address (PA) Structure:** The corresponding PA is formed by concatenating the Physical Page Number (PPN) and the Physical Page Offset (PPO). Importantly, the VPO and PPO are identical, as they represent the offset within a page.

3. **Translation Process:** The VPN is used as an index into the page table to select the relevant Page Table Entry (PTE). The PPN extracted from the PTE is then concatenated with the VPO to create the final PA.

Page Hits and Page Faults

The core difference in handling memory access depends on the PTE's valid bit:

• **Page Hit:** If the valid bit in the PTE is set, the page is cached in DRAM. The MMU uses the PPN from the PTE to form the PA, which is sent to the cache/main memory to retrieve the requested data. This process is handled entirely by hardware.

• **Page Fault (DRAM Cache Miss):** If the valid bit is clear, the requested page is not in physical memory. This triggers an exception that transfers control to a page fault handler in the operating system kernel. The handler selects a victim page, pages the required virtual page from disk into physical memory, updates the PTE, and restarts the faulting instruction. The restarted instruction then proceeds as a hit.

9.6.1 Integrating Caches and VM

Most systems use **physical addressing** for the SRAM caches (L1, L2, L3). The address translation occurs **before** the cache lookup. This simplifies cache design by ensuring that access rights (permissions) are checked during translation and allowing multiple processes to share blocks in the cache.

9.6.2 Speeding Up Address Translation with a TLB

To accelerate the address translation process, the MMU includes a **Translation Lookaside Buffer (TLB)**, which is a small, virtually addressed cache dedicated to storing recently used PTEs.

• The TLB uses the **VPN** to extract a **TLB index (TLBI)** and **TLB tag (TLBT)**.

• A **TLB hit** means the address translation happens quickly within the on-chip MMU.

• A **TLB miss** requires the MMU to fetch the PTE from the L1 cache.

9.6.3 Multi-Level Page Tables

To conserve memory, especially in systems with large address spaces (like 64-bit), page tables are organized hierarchically.

• If a large chunk of the Virtual Address Space is unallocated, the corresponding high-level PTE can be **null**, meaning the lower-level page tables do not exist, saving significant memory.

• The VA is partitioned into multiple VPNs, each indexing a level of the page table hierarchy.

• The TLB is crucial here, as it caches PTEs from all levels of the hierarchy, preventing the multiple memory accesses that would otherwise be required for translation.

9.7 Case Study: The Intel Core i7/Linux Memory System

This section details the specific implementation of VM on the Core i7 processor running Linux, which utilizes a 48-bit virtual address space and 4 KB pages.

9.7.1 Core i7 Address Translation

The Core i7 employs a **four-level page table hierarchy**.

• The physical address of the Level 1 (L1) page table is stored in the **CR3** control register.

• The VPN is divided into four 9-bit chunks, each serving as an offset into one of the four page table levels (L1 through L4).

• Level 1, 2, and 3 PTEs contain a **40-bit PPN** pointing to the next page table. Level 4 PTEs contain the **PPN** of the final physical page.

• PTEs include several permission bits, notably:

    ◦ **R/W (Read/Write):** Controls read/write access.

    ◦ **U/S (User/Supervisor):** Restricts user-mode processes from accessing kernel code/data.

    ◦ **XD (Execute Disable):** Can prevent instruction fetches from certain memory pages, mitigating buffer overflow attacks.

• The MMU manages status bits in the PTEs: the **A (reference) bit** set upon access, and the **D (dirty) bit** set upon write.

9.7.2 Linux Virtual Memory System

Linux views a process's virtual memory as a collection of **areas** (or segments), which are contiguous chunks of allocated virtual memory.

• The kernel uses a **task structure** pointing to an **mm_struct**, which manages the virtual memory state.

• The `mm_struct` contains `pgd` (pointer to the L1 page table) and a list of **vm_area_structs** (area structs) that describe each VM area.

• Area structs define the area bounds (`vm_start`, `vm_end`), permissions (`vm_prot`), and sharing status (`vm_flags`).

• **Page Fault Handling:** When a page fault occurs, the kernel checks:

    1. **Legality of Address:** Does the address fall within the bounds of any defined area struct? If not, a segmentation fault occurs.

    2. **Legality of Access:** Does the process have the correct permissions (read, write, or execute) for that area? If not, a protection exception occurs.

    3. If both checks pass, the kernel handles the fault by paging the necessary data into memory.

9.8 Memory Mapping

**Memory mapping** is the process by which Linux associates a virtual memory area with an object on disk.

Mapping Types

Areas can be mapped to two types of objects:

1. **Regular File:** Mapped to a contiguous section of a disk file. The actual content is only loaded (paged in) by the VM system when the CPU first accesses the page (demand paging).

2. **Anonymous File:** Mapped to an anonymous, kernel-created file of all zeros. Pages in this area are called **demand-zero pages**. When first referenced, the kernel allocates a physical page, overwrites it with zeros, and updates the PTE, without transferring any data from disk.

All allocated VPs are swapped back and forth between physical memory and a **swap file** (or swap space) maintained by the kernel.

9.8.1 Shared Objects Revisited

The memory mapping mechanism controls how objects are shared:

• **Shared Object:** Any writes made by one process to a shared area are visible to all other processes that have mapped that object. Changes are also reflected back to the original object on disk. Only a single copy of the shared object is stored in physical memory, saving resources.

• **Private Object:** Writes are not visible to other processes and are not reflected on disk. Private objects use **copy-on-write**: initially, processes share the object, but the PTEs are flagged as read-only. An attempted write triggers a protection fault, and the kernel creates a new copy of the page for the writing process.

9.8.2 The fork Function Revisited

When `fork` is called, the kernel duplicates the parent's VM structures, flags all pages as **read-only**, and marks the areas as **private copy-on-write**. This ensures that the child receives an identical, but independent, virtual address space.

9.8.3 The execve Function Revisited

`execve` loads a new program into the current process's address space.

1. It **deletes** the existing user areas.

2. It creates new **private copy-on-write areas** for the new program's code, data, BSS, and stack. The code and data areas map to the executable file; BSS, stack, and heap areas are **demand-zero**.

3. It maps shared areas (like `libc.so`) into the shared region.

4. It sets the program counter to the program's entry point (`_start`).

9.8.4 User-Level Memory Mapping with the mmap Function

The `mmap` function allows applications to manually create and delete VM areas.

• `mmap` creates a new VM area and maps a disk file segment (`fd`, `offset`, `length`) to it.

• The `prot` argument sets page permissions (`PROT_READ`, `PROT_WRITE`, etc.).

• The `flags` argument determines mapping type (`MAP_ANON` for demand-zero, `MAP_PRIVATE` for copy-on-write, `MAP_SHARED` for shared).

• The `munmap` function deletes the created virtual memory area.
